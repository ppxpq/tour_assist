# Web 端升级开发文档

## 背景

当前项目已经具备 FastAPI 后端、多 Agent 工作流、RAG 知识库、会话管理、文件上传、SSE 流式输出、高德工具和 12306 MCP 能力，但 Web 端体验仍偏 Demo 化：

- 旅行记录标题直接来自第一条会话内容，概括感不足。
- 用户主要靠纯文本输入，缺少标签、选项、日期等结构化交互。
- Planner 输出以长 Markdown 文本为主，视觉层级弱，行程阅读成本高。
- 内容生成时信息量很大，缺少更适合展示行程的卡片化 UI。

本阶段目标是先提升 Web 端产品感，不急着做小程序、地图、电信业务深度接入。

## 升级目标

把当前 Web 端从“旅游聊天机器人”升级为“出行规划工作台”：

- 旅行记录标题更短、更像真实行程记录。
- 输入方式从纯文本升级为“标签/表单 + 自由补充”。
- 行程结果从长文本升级为结构化、分区、卡片化展示。
- 尽量复用现有后端 API，优先做低风险、高观感收益的改动。

## v1 范围建议

第一阶段建议只做三个核心能力：

1. 标签式出行需求输入。
2. 卡片化行程展示。
3. 智能凝练会话标题。

暂不纳入 v1：

- 小程序。
- 地图路线。
- 登录系统。
- 复杂用户画像。
- 电信业务深度接入。
- 结构化 JSON 全链路输出。

## 功能优先级

| 功能 | 复杂度 | 优先级 | 说明 |
|---|---:|---:|---|
| Planner 输出格式约束 | 中 | P0 | 直接改善最终结果观感 |
| 前端卡片化展示 | 中 | P0 | 让结果不像纯文本聊天 |
| 标签/表单式输入 | 中 | P0 | 产品感提升明显 |
| 旅行记录标题凝练 | 低 | P1 | 实现简单，但可以等 Router/Planner 信息更稳定后做 |
| 结构化 JSON 行程 | 中高 | P1 | 后续地图和小程序需要，但 v1 可以先不强依赖 |
| 地图路线 | 高 | P2 | 值得做，但建议放到第二阶段 |

## 方案一：旅行记录标题凝练

### 现状

当前标题通常来自第一条用户消息截断，例如：

```text
我想去杭州玩 3 天，喜欢...
```

问题：

- 太长。
- 太口语。
- 很难快速区分会话。
- 对车票查询、问答、知识库问答等场景概括不够准确。

### v1 目标

生成类似下面的标题：

```text
杭州 3日 · 美食人文
西安高铁票查询
亲子游目的地推荐
泉州 4日 · 老城美食
```

### 实现建议

第一阶段先用规则生成，不额外调用 LLM：

- 从 Router 结果读取 `intent`、`city`、`days`、`preference`。
- `need_plan`：生成 `{city} {days}日 · {preference}`。
- `need_ticket`：生成 `{departure}到{city}车票查询`。
- `need_answer`：从用户问题截取关键词，生成短标题。
- 兜底：继续使用 `_short_title()`。

### 需要改动

- `core/travel_service.py`
- 可选：`agents/ticket_node.py` 返回更完整的 `departure` 信息

### 待讨论

- 是否允许额外调用一次轻量模型生成标题？ ～～基于规则的关键词提取不成功再调用，你觉得怎么样？
- 标题是否要在最终回复完成后再更新，而不是用户发消息时立即更新？ ～～如果能基于规则提取成功，就立刻更新，然后每次会话都要check一下是否要更新，开销会比较大吗？

## 方案二：标签式出行需求输入

### 现状

用户只能在文本框中自由输入，例如：

```text
我想 5 月底去泉州 4 天，喜欢老城、美食和慢节奏
```

问题：

- 对新用户不友好。
- 很容易漏填城市、日期、天数、偏好。
- 交互看起来像普通聊天机器人。

### v1 目标

在聊天输入区上方增加一个轻量“出行需求面板”：

```text
目的地：[输入框]
出发日期：[日期选择]
天数：[1天] [2天] [3天] [4天] [5天+]
同行人：[独自] [朋友] [情侣] [亲子] [家庭] [银发]
偏好：[美食] [人文] [自然] [摄影] [休闲] [小众] [省钱]
交通：[高铁] [自驾] [公交] [步行] [不确定]
预算：[经济] [舒适] [品质]
补充要求：[自由输入]
```

点击“生成规划”后，前端拼成自然语言，继续调用现有 `/api/chat/stream`。

示例拼接结果：

```text
我想去杭州玩 3 天，2026-07-10 出发，同行人是亲子，偏好美食、人文，交通方式高铁，预算舒适。补充要求：不要太累。
```

### 实现建议

第一阶段不改后端 API：

- 前端新增表单状态。
- 点击标签后写入状态。
- 点击生成后把状态拼成 prompt。
- 仍然走 `sendStreamingChat()`。
- 原始 textarea 保留，用于补充自由需求。

### 需要改动

- `frontend/index.html`
- `frontend/styles.css`
- `frontend/app.js`

### 待讨论

- 需求面板默认展开还是折叠？～～需求面板又是啥
- 是否保留原来的“灵感模板”？～～灵感模版是啥
- 标签是否需要多选？～～肯定要啊。
- 日期选择是否先用原生 `input type="date"`？～～啥意思

## 方案三：Planner 输出格式优化

### 现状

Planner 输出 Markdown 长文本，内容可读但视觉层级弱。

### v1 目标

先约束 Planner 输出稳定的 Markdown 结构，便于前端渲染：

```markdown
# 杭州 3 日行程

## 行程概览

- 主题：美食 + 人文 + 慢节奏
- 强度：中等
- 适合人群：亲子 / 家庭
- 交通方式：高铁 + 市内公共交通

## Day 1 · 西湖与老城

日期：2026 年 07 月 10 日
天气：晴转多云，24-32℃
主题：经典景点 + 老城初体验

### 上午

- 地点：西湖
- 推荐理由：经典杭州印象，适合轻松开始行程
- 建议停留：2 小时
- 交通建议：从酒店打车或地铁前往

### 下午

...

### 晚上

...

### 餐饮建议

...

## 注意事项

...

## 预算建议

...
```

### 实现建议

先不要求 Planner 输出 JSON，避免一次性改动过大。

第一阶段：

- 强化 `agents/planner_node.py` 中的 Prompt。
- 要求固定标题层级和字段名。
- 前端继续使用 Markdown 渲染，但针对行程标题、Day 标题、时间段标题做卡片样式。

第二阶段：

- 再让 Planner 输出 Markdown + JSON 双格式。
- JSON 用于地图、小程序、结构化卡片。

### 需要改动

- `agents/planner_node.py`
- `frontend/app.js`
- `frontend/styles.css`

### 待讨论

- 是否保留 Markdown 为主要输出格式？
- 是否现在就引入 JSON？
- 卡片样式要偏“旅游产品”还是偏“工作台/效率工具”？

## 方案四：前端卡片化展示

### 现状

前端已有简易 Markdown 渲染：

- `#` / `##` / `###`
- 列表
- 代码块
- 加粗

但所有内容都在普通气泡内，缺少行程卡片感。

### v1 目标

增强 Markdown 渲染样式，不重写成复杂富文本：

- `# 行程标题`：大标题。
- `## Day X`：渲染为 Day 卡片分区。
- `### 上午/下午/晚上`：渲染为时间段块。
- `## 注意事项`、`## 预算建议`：渲染为强调信息块。

### 实现建议

低风险方案：

- 保持 `renderMarkdown()` 逻辑。
- 主要通过 CSS 增强 `.bubble h1/h2/h3/ul/li` 样式。
- 对包含 `Day` 的 `h2` 添加特殊 class。

中风险方案：

- 在 `renderTextBlock()` 中识别标题文本。
- 输出更明确的 HTML 结构，例如：

```html
<section class="itinerary-day">
  <h2>Day 1 · 西湖与老城</h2>
</section>
```

### 需要改动

- `frontend/app.js`
- `frontend/styles.css`

### 待讨论

- 是否要支持折叠 Day 卡片？
- 是否要固定右侧摘要目录？
- 是否要在结果顶部增加“复制行程 / 导出”按钮？

## 推荐开发顺序

建议按以下顺序推进：

1. Planner 输出格式优化。
2. 前端卡片化展示。
3. 标签式出行需求输入。
4. 旅行记录标题凝练。

原因：

- Planner 和展示决定最终观感，是当前最明显短板。
- 标签输入会影响入口交互，但不解决结果丑的问题。
- 标题凝练较简单，可以最后补。

## 验收标准

v1 完成后，应满足：

- 用户不需要纯手打完整需求，也能通过标签快速生成规划。
- 行程结果视觉上不再像一整段聊天文本。
- 每天行程、时间段、餐饮、注意事项、预算建议有清晰层级。
- 会话标题能快速识别城市、天数和主题。
- 现有普通聊天、车票查询、知识库问答不被破坏。

## 风险与注意事项

- 过早引入 JSON 可能导致 Planner 输出不稳定，需要更多解析和兜底。
- 只美化前端但不约束 Planner 输出，卡片化效果会不稳定。
- 标签输入不要替代自由输入，否则灵活性下降。
- 生成过程中不要锁死滚动、分栏拖拽等基础浏览交互。
- 后续如果做地图或小程序，最好再推进结构化行程数据。
