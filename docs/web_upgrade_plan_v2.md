# Web 端升级开发文档 v2

## 目标

本轮先聚焦两个方向：

1. 旅行记录标题凝练与动态更新。
2. 聊天形态下的标签/结构化输入增强。

核心原则：

- 不把聊天替换成表单，而是在聊天里增加结构化辅助。
- 标签不是限制用户表达，而是帮助用户更快描述需求。
- 用户随时可以修改行程条件，不应该被要求重新开会话。
- 交通能力需要细分，不能只用一个“高铁/自驾”标签概括。

## 方案一：旅行记录标题凝练与动态更新

### 问题

当前旅行记录标题主要来自第一条用户消息截断。

这会导致两个问题：

1. 标题不够凝练。
2. 会话中途修改需求时，标题不会自然反映最新状态。

典型场景：

```text
用户：我想去杭州玩 3 天，喜欢美食和人文
系统标题：杭州 3日 · 美食人文

用户：改成 4 天吧
系统标题应更新为：杭州 4日 · 美食人文
```

用户不应该因为修改天数、目的地、偏好等条件而重新开会话。

### 目标

标题应是“当前会话出行状态”的摘要，而不是第一句话摘要。

示例：

```text
杭州 4日 · 美食人文
泉州 3日 · 老城慢游
南京到西安车票
亲子游目的地推荐
```

### 核心字段

标题生成主要参考以下字段：

- `intent`：规划 / 问答 / 车票 / 闲聊
- `city`：目的地
- `days`：天数
- `start_date`：出发日期
- `preference`：旅行偏好
- `traveler_type`：同行人类型
- `budget_level`：预算层级
- `departure`：出发地
- `arrival_transport`：出发地到目的地交通方式
- `local_mobility`：目的地内交通/机动能力

其中 v1 必须支持：

- `intent`
- `city`
- `days`
- `preference`
- `departure`

其他字段可以先从标签输入拼接到用户 prompt 中，后续再结构化进入 Router 状态。

### 更新策略

标题不是只在新会话生成一次，而是每轮消息后检查是否需要更新。

但也不应每轮无意义刷新。建议规则：

更新标题的条件：

- 当前标题是 `新会话`。
- 当前标题仍是原始用户输入截断。
- 本轮明确修改了核心字段，例如：
  - 改成 4 天
  - 换成苏州
  - 出发地改上海
  - 这次带老人
  - 预算改舒适
- 当前会话前几轮信息逐步补齐，标题可以变得更准确。

不更新标题的情况：

- 用户只是追问某个景点细节。
- 用户问标签含义。
- 用户要求解释某段行程。
- 用户只是表达感谢、确认、闲聊。

### 生成方式

采用：

```text
规则优先 + 低质量兜底 LLM
```

规则能生成时，直接生成标题。

规则失败或标题质量低时，再调用轻量模型生成 8-12 字标题。

### 规则示例

行程规划：

```text
{city} {days}日 · {preference}
```

如果有同行人：

```text
{city} {days}日 · {traveler_type}
```

如果偏好和同行人都有：

```text
{city} {days}日 · {preference}
```

车票查询：

```text
{departure}到{city}车票
```

问答：

```text
{city}旅行问答
```

兜底：

```text
使用模型凝练为 8-12 字标题
```

### 实现建议

v1 可以不新增复杂数据库表，先在现有会话更新逻辑中做：

1. 每轮 Graph 执行后读取 Router/节点输出里的核心字段。
2. 用规则生成候选标题。
3. 判断候选标题是否比当前标题更好。
4. 如果更好，调用 `SessionStore.update_session()` 更新标题。

后续可演进：

- 增加 `session_context` 表，保存当前目的地、天数、偏好等结构化状态。
- 用户修改条件时，不只更新标题，也更新结构化状态。

### 待确认

- v1 是否需要引入 `session_context`，还是先只更新标题？
- 标题是否展示出发日期？
- 标题是否展示预算/同行人，还是只展示城市、天数、偏好？

## 方案二：聊天形态下的标签/结构化输入增强

### 问题

当前输入方式基本是纯文本。

纯文本的优点是灵活，但问题是：

- 用户容易漏填目的地、天数、日期、同行人等关键信息。
- 对新用户没有引导。
- 产品看起来像普通聊天机器人。
- 出行交通需求很复杂，单靠“高铁/自驾”标签不够。

### 核心原则

保留聊天形态，不改成纯表单。

用户可以自由组合以下方式：

1. 直接打字。
2. 选择标签后发送。
3. 选择标签 + 手动补充。
4. 点击灵感模板快速填入。
5. 对标签含义进行提问。

标签输入只是“需求构建器”，不是替代聊天。

### 页面结构建议

聊天输入区可拆成三层：

```text
快捷需求标签区
自由输入区
发送操作区
```

右侧可以继续保留“灵感模板”，但后续可以弱化为“示例场景”。

### 字段设计

#### 1. 基础信息

目的地：

```text
[输入框：想去哪里？]
```

出发日期：

```text
[日期选择：YYYY-MM-DD]
```

天数：

```text
[1天] [2天] [3天] [4天] [5天+] [自定义]
```

补充要求：

```text
[自由输入：还有什么特别要求？]
```

#### 2. 同行人

支持多选：

```text
[独自] [朋友] [情侣] [亲子] [家庭] [老人] [同事]
```

说明：

- `亲子` 影响节奏、景点安全性、餐饮便利度。
- `老人` 影响步行距离、休息频率、无障碍、医疗/服务点。
- `家庭` 可能同时包含亲子、老人，需要允许多选。

#### 3. 偏好

支持多选：

```text
[美食] [人文] [自然] [摄影] [休闲] [小众] [省钱] [购物] [夜游]
```

说明：

- 多选后用于拼接 prompt。
- Planner 需要根据多个偏好做权衡，而不是机械堆景点。

#### 4. 预算

建议单选：

```text
[经济] [舒适] [品质]
```

标签解释：

- `经济`：优先控制总花费，更多使用公共交通、平价餐饮、免费/低价景点。
- `舒适`：在预算和体验之间平衡，减少过度折腾，适当选择更省力交通和餐饮。
- `品质`：更重视体验质量，减少排队和转场，可安排更高品质餐饮、住宿或体验项目。

#### 5. 交通信息

交通需要拆成两类：

```text
出发地到目的地交通
目的地内交通/机动能力
```

不能只做一个笼统“交通方式”标签。

##### 5.1 出发地到目的地交通

字段：

```text
出发地：[输入框]
到达方式：[高铁] [火车] [飞机] [自驾] [大巴] [不确定]
```

用途：

- 影响首日/末日行程强度。
- 影响是否需要查询车票。
- 影响到达站/机场到酒店的交通建议。
- 影响预算估算。

示例 prompt：

```text
我从南京出发，倾向高铁到杭州。
```

##### 5.2 目的地内交通/机动能力

字段：

```text
当地交通偏好：
[公共交通] [打车] [自驾/租车] [骑行] [步行友好] [少走路] [无障碍优先]
```

建议支持多选。

解释：

- `公共交通`：优先地铁/公交，减少打车成本。
- `打车`：减少换乘和步行，适合亲子/老人/赶时间。
- `自驾/租车`：适合郊区、多点位、跨城路线。
- `骑行`：适合城市慢游、绿道、湖区、短距离点位。
- `步行友好`：用户可以接受较多步行，路线可安排街区漫游。
- `少走路`：限制步行距离，景点之间尽量打车或选择集中区域。
- `无障碍优先`：考虑轮椅、老人、婴儿车、台阶少、休息点多。

用途：

- 影响多个景点之间的路线安排。
- 影响每日景点数量。
- 影响每段交通建议。
- 影响 Planner 对体力消耗的判断。

示例 prompt：

```text
当地交通希望以打车为主，老人同行，少走路。
```

##### 5.3 体力/步行能力

交通标签还需要体现“人能不能走”的问题。

可以增加字段：

```text
步行能力：
[轻松少走] [普通强度] [可以多走] [可骑行] [需要无障碍]
```

也可以先并入“当地交通偏好”里。

v1 推荐先并入“当地交通偏好”，减少 UI 复杂度。

### 标签解释能力

用户可能会问：

```text
经济和舒适有什么区别？
什么叫少走路？
无障碍优先是什么意思？
步行友好是啥？
```

系统必须能回答这些问题，而不是强制进入行程规划。

实现建议：

#### 前端层

标签增加 `title` 或说明浮层。

第一阶段可先用浏览器原生 tooltip：

```html
<button title="经济：优先控制总花费，更多使用公共交通和平价餐饮">经济</button>
```

后续再做自定义说明浮层。

#### 后端层

Router 需要能把标签解释问题识别为 `need_answer`。

可选增强：

- 增加本地 FAQ。
- 对包含“什么意思 / 区别 / 是啥 / 怎么选”的问题优先回答标签解释。

### Prompt 拼接策略

前端选择标签后，仍然拼接成自然语言发送给现有接口。

示例：

```text
我想去杭州玩 4 天，2026-07-10 出发。
出发地：南京；到达方式：高铁。
同行人：家庭、老人。
偏好：美食、人文、休闲。
预算：舒适。
当地交通偏好：打车、少走路、无障碍优先。
补充要求：老人腿脚一般，每天不要安排太满。
```

这样后端现有 Router/Planner 可以继续工作。

后续如果需要地图、小程序或更稳定的结构化能力，再把这些字段作为 JSON 传给后端。

### v1 交互建议

默认展示一个轻量标签区，不做过重表单。

建议布局：

```text
目的地 / 日期 / 天数
同行人
偏好
预算
出发地与到达方式
当地交通偏好
自由输入
发送
```

标签区可以折叠，但 v1 建议默认展开，减少用户不知道有这个功能的问题。

### 灵感模板

保留灵感模板。

原因：

- 标签不能覆盖所有表达。
- 新用户需要示例。
- 用户对标签不理解时，可以从模板获得提示。
- 后续可以把灵感模板变成“示例场景”：
  - 亲子轻松游
  - 老人少走路
  - 周末高铁游
  - 省钱学生游
  - 城市美食慢游

### 待确认

- 交通信息是否 v1 就拆成“出发地到目的地”和“当地交通偏好”？
- 当地交通偏好是否多选？
- 是否单独做“步行能力”，还是先并入当地交通偏好？
- 标签解释先用原生 tooltip，还是做自定义说明浮层？
- 标签区默认展开还是可折叠但默认展开？

## 初步开发顺序

建议先做方案二，再做方案一。

原因：

- 标签式输入会改变用户生成 prompt 的方式，也会提供更多标题生成字段。
- 标题凝练可以直接利用标签输入形成的结构化信息。

开发顺序：

1. 实现标签/结构化输入 UI。
2. 将标签输入拼接成自然语言 prompt，继续调用现有 `/api/chat/stream`。
3. 增加标签 tooltip。
4. 增强 Router 对标签解释问题的识别。
5. 实现标题动态更新。

## 当前实现状态

已完成：

- 标签/结构化输入 UI。
- 标签输入拼接为自然语言 prompt，继续调用现有 `/api/chat/stream`。
- 预算、交通、同行人等标签的基础 tooltip。
- Router 对标签解释问题的本地识别与直接回复。
- 会话标题基于规则动态更新，支持用户中途修改目的地、天数、偏好等核心条件。
- Planner 输出格式约束，按行程概览、Day、上午/下午/晚上、注意事项、预算建议/安排组织。
- 前端 Markdown 初步卡片化展示，Day、时间段、概览、注意事项、预算区域有独立视觉分区。

待继续：

- 更完整的标题状态管理，例如独立 `session_context`。
- 标签解释自定义浮层。
- 地图路线或小程序所需的结构化 JSON 输出。

## v1 验收标准

- 用户可以纯聊天，也可以点标签生成需求。
- 偏好、同行人、当地交通偏好支持多选。
- 交通被拆分为“出发地到目的地”和“目的地内交通/机动能力”。
- 用户可以表达“少走路、可骑行、租车、自驾、无障碍”等能力差异。
- 用户询问标签含义时，系统能回答，而不是强制规划。
- 会话标题可以随目的地、天数等核心需求变化而更新。
